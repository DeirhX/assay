# Assay

*Weigh the metal, not the hype — test every number before you trust it.*

Assay is a single-user portfolio research and rebalancing workbench. Like a
metallurgical assay tests whether a bar is really gold, it tests whether the
numbers behind a position — price, market cap, multiples, target weights — hold
up before you act on them.

It runs **locally, on your own machine, against your own private holdings**. It
is not multi-user, not a hosted service, and not a robo-advisor. Research never
places trades on its own; the only surface that can touch an order is the opt-in,
paper-first **Trade desk**, which is off by default.

---

## What you get

- **Verified numbers.** Live deep dives on any ticker — price, momentum, market
  cap, P/E (ttm/fwd), P/S, revenue, margins, share count — pulled from Yahoo and
  cross-checked against SEC EDGAR (and a third source if configured). Mismatches
  are flagged, not smoothed over.
- **A standing rebalance plan.** Current NAV weights against your target bands,
  with risk, tax-lot, and what-if tooling to pressure-test a basket before you
  place it.
- **Segment & peer dashboards.** A whole peer universe (e.g. semiconductors)
  ranked and cross-joined against what you already own.
- **A Deep Research pipeline.** Perplexity report capture, source extraction, a
  local review gate, and draft target-model proposals — none of which ever create
  trades.
- **Facts kept apart from judgement.** Fetched numbers and your written thesis
  live separately; re-pulling data never clobbers your reasoning.
- **An optional Trade desk.** Paper-first order placement through Interactive
  Brokers, one confirmed order at a time.

## Run it

You only need two things installed:

- **Python 3** (standard library only — nothing to `pip install` for the app).
- **Node.js + npm** (to build the web UI once).

Then, from the repo root:

```powershell
git submodule update --init                          # optional: your private holdings
npm install && npm run build                          # build the web UI (once, and after UI updates)
$env:SEC_USER_AGENT = "assay research (you@example.com)"
py -3 tools/serve.py                                  # starts the app
```

Open **http://127.0.0.1:6060** in your browser. The app binds to localhost only
and runs live network pulls on demand, so don't expose it to the internet.

> First run without any extra config still works — you just get the core research
> features. The portfolio views are empty until you add holdings, and a few
> features stay dormant until you give the app the matching config below.

## Configure it — what unlocks each feature

Assay reads its configuration from three places:

1. **`secrets.env`** in the repo root (gitignored). Plain `KEY=VALUE` lines,
   loaded automatically when the app starts. This is the easiest place for your
   keys — see the example below.
2. **Environment variables** in your shell. Same names; these win over
   `secrets.env` if both are set.
3. **The in-app Setup tab** at **http://127.0.0.1:6060/?view=setup**, which guides
   you through the pieces that can't be a simple key (LLM logins, the Perplexity
   session, and IBKR credentials, which it writes to `tools/secrets.env` for you).

| Feature | What it needs | Required? |
| --- | --- | --- |
| Your own portfolio & targets | The private `data/` submodule (`git submodule update --init`, needs repo access) | For any portfolio view |
| SEC-verified fundamentals | `SEC_USER_AGENT` identifying you to the SEC | Strongly recommended |
| Third market-data opinion | `FMP_API_KEY` (Financial Modeling Prep) | Optional |
| AI analyst notes | A local LLM CLI — `claude` or `cursor-agent` — installed and logged in | Optional |
| Deep Research pipeline | A Perplexity Pro login (set up via the Setup tab) | Optional |
| Trade desk (paper) | IBKR Client Portal Gateway running + `IBKR_TRADING_ENABLED=1` | Optional |
| Trade desk (live orders) | The above **plus** `IBKR_ALLOW_LIVE=1`, after paper validation | Optional, high-risk |

### Your portfolio data

Your holdings, NAV, and targets live in a **private `data/` git submodule** — the
public code contains no account data. Run `git submodule update --init` (you need
access to that private repo). Without it the app still starts; the portfolio and
target views are simply empty.

### SEC-verified fundamentals — `SEC_USER_AGENT`

The SEC's fair-access API asks callers to identify themselves. Set
`SEC_USER_AGENT` to a short string with your contact info, e.g.
`assay research (you@example.com)`. Without it, SEC cross-checks may be rate-limited
or rejected, which weakens the data-trust panel.

### Third market-data opinion — `FMP_API_KEY`

A Financial Modeling Prep API key adds a third independent source alongside Yahoo
and SEC, so disagreements are easier to catch. Optional — leave it unset and Assay
just uses two sources.

### AI analyst notes — a local LLM CLI

Per-ticker analyst notes are generated by a local command-line LLM. Install and
log into **`claude`** (preferred) or **`cursor-agent`** (fallback), then use the
Setup tab's smoke check to confirm it actually works — having the binary on your
`PATH` is not enough; it must be authorized.

### Deep Research pipeline — Perplexity Pro

The Deep Research pipeline drives Perplexity through a real browser using your
**Perplexity Pro** quota. In the Setup tab, click **Set up Perplexity login**: a
browser window opens against a persistent profile; sign in once and later runs
reuse that session.

### Trade desk — Interactive Brokers

The Trade desk is the only part that can place orders, and it's **off by default**.
It talks to the IBKR **Client Portal Gateway** (a small Java program you run and
log into yourself, with 2FA). To enable it:

- Set `IBKR_TRADING_ENABLED=1` to switch on the desk (paper accounts only).
- Live, real-money placement stays locked until you **also** set `IBKR_ALLOW_LIVE=1`
  — and only do that after validating on a paper account.
- Optional: `IBKR_GATEWAY_BASE` (defaults to `https://localhost:5000/v1/api`) and
  `IBKR_TRADE_ACCOUNT_ID` (defaults to preferring a paper `DU…` account).

The Setup tab can capture your IBKR Flex credentials and write them to
`tools/secrets.env` for you. See **Trade desk safety** below before placing anything.

### Example `secrets.env`

Create a file named `secrets.env` in the repo root (it's gitignored):

```ini
SEC_USER_AGENT=assay research (you@example.com)
FMP_API_KEY=your-fmp-key-here
# Trading is off unless you uncomment these:
# IBKR_TRADING_ENABLED=1
# IBKR_ALLOW_LIVE=1
```

Never commit `secrets.env`, API keys, IBKR credentials, browser profiles, or raw
account exports.

## Trade desk safety

The desk is deliberately hard to fire by accident:

- Trading is refused entirely unless `IBKR_TRADING_ENABLED` is set.
- A basket must be **previewed before it can be placed** — the preview returns a
  token the place step must echo, so a basket that changed after preview is
  rejected.
- Orders are re-derived server-side from that token-bound basket; nothing is
  trusted from the browser.
- **Live (non-paper) placement stays locked** until `IBKR_ALLOW_LIVE` is also set.
  Paper accounts are recognised by their `DU` prefix.

Validate on paper before ever enabling live:

- [ ] Gateway running and logged in; the Trade tab shows *Paper account DU…*.
- [ ] Stage a small basket, **Preview**, and confirm the share counts and IBKR
      margin/commission look sane.
- [ ] **Place** it on paper; confirm the orders appear in IBKR, then cancel or let
      them fill.
- [ ] Re-preview after the fill and confirm positions moved as expected.

## Privacy & your data

Assay is local-first by design. The public repo carries **no** holdings, NAV, or
P/L — those come from your private `data/` submodule and are never committed. Keep
credentials, raw IBKR exports, full account snapshots, browser profiles, and
`secrets.env` out of version control.

placement requires a second flag (`IBKR_ALLOW_LIVE`) on top. Order execution is
inherently risky: prices move, orders can partially fill or be rejected, and a
mistaken basket can lose real money. **You alone are responsible for every order

this software. Validate on a paper account first. Use at your own risk.
